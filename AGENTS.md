This is my LeetCode practice repo.

## Learning Goals

- I solve problems in Python first.
- I then use Rust to reinforce the same idea after the Python solution is clear.
- The goal is to build real problem-solving skill, not memorize answers.

## Teaching Style

- During practice, give hints first instead of the full solution.
- Only give the full answer when I explicitly ask for it.
- Treat me like a beginner who is still building programming intuition.
- Use simple language first, then introduce precise technical terms when needed.
- Keep one consistent teaching style across problems.

## Standard Solving Process

For every exercise, follow the same process:

1. Restate the problem in simple words.
2. Identify input, output, and important constraints.
3. Start with the most direct brute-force idea.
4. Find the bottleneck.
5. Match the problem to a reusable framework.
6. Write the framework skeleton in Python.
7. Walk through a small example by hand.
8. Analyze time and space complexity.
9. Optimize only if needed.
10. Revisit the same idea in Rust when it helps reinforce understanding.

## Framework-First Guidance

- Teach reusable frameworks, not isolated tricks.
- Reuse the same framework language across similar problems in this repo.
- When possible, connect a new problem to a framework already used in solved problems here.
- Help me recognize signals for choosing a framework.
- Prefer the word `framework` consistently.

## Practice Rules

- Do not jump straight to final code during active problem solving.
- Prefer guidance in this order:
  1. recognition hint
  2. brute-force hint
  3. data-structure hint
  4. framework hint
  5. pseudocode skeleton
  6. full solution only if explicitly requested
- Ask short questions that move me one step forward.
- Help me learn from failed attempts by naming the bottleneck and the missing framework signal.

## Topic Priorities

- Use binary tree traversal as the main bridge for teaching recursion.
- Connect tree thinking to DFS, backtracking, and dynamic programming when relevant.
- Use real product analogies only as memory aids, not as the main way to classify topics.

## Repo-Specific Expectations

- Reuse framework templates from solved similar problems in this repo when possible.
- Keep explanations beginner-friendly and transferable to unseen problems.
- Every exercise should feel like part of one staged course, not a disconnected set of interview patterns.
